# ⚽ ניחושי מונדיאל 2026

אתר ניחושים לשלבי הנוקאאוט של מונדיאל 2026 — מהשמינית ועד הגמר.
משתתפים נרשמים עם מייל בלבד, מנחשים מי עולה הלאה ומה תהיה תוצאת 90 הדקות,
וצוברים נקודות לפי שיטת ניקוד שמכפילה ככל שהטורניר מתקדם.

## הפעלה מקומית

```bash
npm install
npm start
```

ואז פותחים בדפדפן: <http://localhost:3000>

(אופציונלי) למילוי שמינית הגמר במשחקים לדוגמה: `npm run seed` — ואז עורכים את
זוגות הנבחרות בפאנל הניהול ברגע שהם נקבעים.

## איך נכנסים

1. מזינים כתובת מייל ולוחצים "שלחו לי קוד".
2. **במצב הדגמה** הקוד בן 6 הספרות מוצג ישירות על המסך (וגם בלוג השרת).
3. מזינים את הקוד → מחוברים. בכל כניסה/יציאה נדרש קוד אקראי חדש.

## ניהול (אדמין)

המייל שמוגדר כאדמין (כברירת מחדל `uv.levari@gmail.com`) רואה טאב **ניהול** ובו:

- **הוספת משחקים** לכל שלב (שמינית / רבע / חצי / גמר) עם מועד פתיחה.
  אחרי מועד הפתיחה הניחושים ננעלים אוטומטית.
- **הזנת תוצאות** — בוחרים מי עלה הלאה ומזינים את תוצאת 90 הדקות.
  ברגע השמירה מחושב הניקוד מחדש לכל המשתתפים, וטבלת הדירוג מתעדכנת.
- **ביטול/מחיקה** של תוצאה או משחק במקרה של טעות.

> **עדכון אחת ל-12 שעות:** בודקים את התוצאות הרשמיות (למשל בגוגל) ומזינים אותן
> בפאנל הניהול. זה לוקח דקה ומעדכן את הניקוד של כולם.

## עדכון תוצאות אוטומטי (API)

במקום הזנה ידנית, אפשר לחבר את **API-Football (API-Sports)** שמושך תוצאות לבד.
היתרון: ה-API מחזיר בנפרד את **תוצאת 90 הדקות** ואת **מי שעלה** (כולל פנדלים) —
בדיוק מה ששיטת הניקוד צריכה.

- מגדירים `FOOTBALL_API_KEY` (מפתח חינמי מ-<https://dashboard.api-football.com>).
- בפאנל הניהול מקשרים כל משחק ל"מזהה משחק ב-API".
- המערכת בודקת תוצאות כל `POLL_MINUTES` דקות (ברירת מחדל 10) ומעדכנת ניקוד אוטומטית.
- בלי מפתח — הכל עובד בהזנה ידנית. ראו פירוט מלא ב-[DEPLOY.md](DEPLOY.md).

## שיטת הניקוד

| שלב | זהות העולה | תוצאה מדויקת | בינגו מושלם |
|-----|:---:|:---:|:---:|
| שמינית הגמר (×1) | 2 | 3 | 5 |
| רבע הגמר (×2) | 4 | 6 | 10 |
| חצי הגמר (×3) | 6 | 9 | 15 |
| הגמר (×4) | 8 | 12 | 20 |

- **זהות העולה בלבד** — פגעתם בנבחרת שעלתה, אך לא בתוצאת 90 הדקות.
- **תוצאה מדויקת ב-90 דקות** — ניחשתם בדיוק את התוצאה (למשל 1–1) לפני ההארכה.
- **בינגו מושלם** — פגעתם גם בעולה וגם בתוצאה המדויקת.

ההכפלה לפי שלב נועדה לאפשר למי שמאחור לפתוח פער ולשמור על מתח עד הרגע האחרון.

## הגדרות (משתני סביבה)

ראו `.env.example`. העיקריים:

| משתנה | ברירת מחדל | תיאור |
|-------|-----------|-------|
| `PORT` | `3000` | פורט השרת |
| `DEMO_MODE` | `true` | קוד OTP על המסך. `false` = דורש שילוב מייל אמיתי |
| `ADMIN_EMAILS` | `uv.levari@gmail.com` | מיילים של מנהלים, מופרדים בפסיק |
| `DB_PATH` | `./mondial.db` | נתיב קובץ מסד הנתונים |

## פריסה לאינטרנט (Render / Railway וכו')

1. העלו את הקוד ל-GitHub.
2. צרו שירות Web חדש, פקודת הפעלה: `npm start`.
3. הגדירו `ADMIN_EMAILS` למייל שלכם.
4. **חשוב לשמירת נתונים:** חברו דיסק קבוע (Persistent Disk) והגדירו
   `DB_PATH` לתיקייה שעליו (למשל `/data/mondial.db`), אחרת מסד הנתונים
   יתאפס בכל פריסה מחדש.

## מבנה הפרויקט

```
server.js     — שרת Express + כל ה-API
db.js         — סכמת SQLite
scoring.js    — חישוב הניקוד
config.js     — הגדרות, שלבים, מכפילים
seed.js       — מילוי משחקי דוגמה (אופציונלי)
public/       — ה-Frontend (HTML/CSS/JS, בעברית RTL)
```

## מעבר לשליחת מייל אמיתית (במקום מצב הדגמה)

מצב ההדגמה מציג את הקוד על המסך. למייל אמיתי:
1. התקינו ספריית מייל (למשל `nodemailer`).
2. ב-`server.js`, בנתיב `/api/auth/send-code`, במקום להחזיר `demoCode`
   שלחו את הקוד במייל.
3. הגדירו `DEMO_MODE=false`.
